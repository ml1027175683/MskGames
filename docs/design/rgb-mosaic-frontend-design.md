# RGB 马赛克资产游戏前端设计文档

## 1. 设计目标

本文档描述当前网页端 MVP 的前端设计。目标是让后续开发者能理解当前结构、状态流转、领域规则和测试边界。

当前设计只面向网页前端，不包含 Tauri、Electron、后端 API 或 Steam SDK。

## 2. 技术边界

当前技术栈：

1. Vite。
2. React。
3. TypeScript。
4. CSS。
5. Vitest。
6. Testing Library。
7. localStorage。

明确不包含：

1. 桌面壳。
2. 服务端接口。
3. 数据库。
4. 账号系统。
5. 第三方平台 SDK。

## 3. 页面结构

应用包含三个顶层页面。

### 3.1 挖矿页

职责：

1. 展示自动挖矿状态。
2. 展示总挖矿数量。
3. 展示最近 10 条产出记录。

### 3.2 库存页

职责：

1. 展示色块库存。
2. 展示画作库存。
3. 展示当前选中色块或画作详情。
4. 承载画作鉴定、重命名、删除和继续创作入口。

### 3.3 画布页

职责：

1. 新建画作。
2. 选择未鉴定画作。
3. 选择可用色块。
4. 编辑 `16x16` 像素矩阵。
5. 展示画布缩放控件。

## 4. 核心状态

当前核心状态集中在 `App.tsx`。

主要状态：

1. `activePage`：当前顶层页面。
2. `inventory`：色块库存。
3. `selectedColor`：当前选中色块。
4. `minedCount`：挖矿总数。
5. `miningRecords`：最近挖矿记录。
6. `artworks`：作品列表。
7. `assets`：资产列表。
8. `activeWorkId`：画布正在编辑的作品。
9. `selectedArtworkId`：库存详情正在查看的作品。
10. `certificationScan`：当前鉴定扫描状态。
11. `message`：全局提示消息。

## 5. 状态分离原则

`activeWorkId` 和 `selectedArtworkId` 必须分离。

原因：

1. 用户在库存页查看作品，不代表正在编辑该作品。
2. 用户进入画布页，不应该自动创建或自动编辑作品。
3. 删除当前编辑作品时，只影响画布编辑状态，不应该破坏库存列表。
4. 已鉴定作品从画作库存移入资产库，只能作为资产查看。

## 6. 领域规则

### 6.1 挖矿

`mineColor()` 负责产出颜色。

规则：

1. 先按稀有度概率抽取等级。
2. 再从该等级代表色池抽取 RGB。
3. 产出后通过 `addColorToInventory()` 叠加库存。

### 6.2 色块叠放

相同 RGB 使用 `colorKey()` 判断是否相同。

规则：

1. 相同 RGB 数量加 1。
2. 新 RGB 插入库存。
3. 色块数量可以因填色减少。
4. 覆盖像素时旧颜色返还库存。

### 6.3 填色

`fillPixel()` 处理像素填充。

规则：

1. 必须存在当前编辑作品。
2. 必须选中一个色块。
3. 选中色块库存数量必须大于 0。
4. 点击同色像素不消耗库存。
5. 覆盖异色像素时消耗新颜色并返还旧颜色。
6. 每次填色更新作品像素和更新时间。

### 6.4 作品生命周期

作品状态：

1. `draft`：未鉴定草稿，可编辑。
2. `certified`：已鉴定资产，不可编辑。
3. `archived`：预留状态。

当前只使用 `draft` 和 `certified`。

### 6.5 鉴定

`certifyWork()` 处理鉴定。

规则顺序：

1. 如果正在扫描，阻止重复操作。
2. 如果作品已鉴定，阻止鉴定。
3. 如果作品未完成，阻止鉴定。
4. 计算完整像素矩阵哈希。
5. 如果哈希已存在，阻止鉴定且不进入扫描。
6. 如果哈希唯一，进入扫描动画。
7. 扫描完成后生成资产并锁定作品。

重复校验必须发生在扫描开始前，避免用户看到假的扫描过程。

## 7. 本地持久化

当前使用 `saveKey = 'rgb-mosaic-save-v1'`。

保存字段：

1. `activeWorkId`。
2. `artworks`。
3. `assets`。
4. `inventory`。
5. `minedCount`。
6. `selectedArtworkId`。

读取规则：

1. 没有存档时创建默认游戏。
2. 存档损坏时回退默认游戏。
3. 缺少内置哥布林时自动补回。
4. 缺少内置皮卡丘时自动补回。
5. 当前编辑作品不存在时重置为未选择。

### 7.1 存档导入导出

`App` 负责生成当前 `SavedGame` 快照并导出 JSON。

导入流程：

1. 读取用户选择的 JSON 文件。
2. `JSON.parse()` 解析内容。
3. 使用 `isSavedGame()` 校验结构。
4. 使用 `normalizeSavedGame()` 补齐内置作品和安全选中项。
5. 同步更新 React state。
6. 由现有保存副作用写回 `localStorage`。

## 8. 组件职责

### 8.1 `App`

职责：

1. 管理全局状态。
2. 执行挖矿循环。
3. 处理持久化。
4. 处理作品创建、选择、填色、鉴定、重命名和删除。
5. 按页面渲染子组件。

### 8.2 `InventoryPage`

职责：

1. 展示色块库存、画作库存和资产库切换。
2. 渲染库存工具栏。
3. 根据当前 tab 渲染色块网格、画作网格或资产网格。
4. 渲染右侧详情面板。
5. 色块库存支持完整图鉴和仅可用颜色切换。

### 8.3 `CanvasPage`

职责：

1. 展示新建画作入口。
2. 展示未鉴定作品下拉选择。
3. 展示可用色块列表。
4. 展示像素画布和缩放控件。
5. 支持在可用色块和完整代表色图鉴之间切换。

### 8.4 `ArtworkDetail`

职责：

1. 展示当前画作详情。
2. 展示作品操作菜单。
3. 处理重命名表单。
4. 处理删除确认。
5. 展示鉴定按钮和继续创作按钮。
6. 展示资产信息。

### 8.5 `CertificationScanModal`

职责：

1. 展示扫描动画。
2. 展示鉴定结果。
3. 提供查看资产详情和关闭结果入口。

### 8.6 `AssetCard`

职责：

1. 展示已鉴定资产缩略图。
2. 展示资产编号、像素指纹摘要和鉴定时间。
3. 点击后切换资产详情。

### 8.7 `AssetDetail`

职责：

1. 展示资产预览。
2. 展示资产编号、像素指纹、创建者、拥有者、鉴定时间和关联作品。
3. 保持只读，不提供编辑、删除和重命名入口。

## 10. 当前技术债

当前 MVP 为了快速验证，把较多逻辑集中在 `App.tsx`。

后续建议拆分：

1. `domain/game.ts`：作品、填色、鉴定领域操作。
2. `storage/localGameStorage.ts`：本地存档读写。
3. `components/inventory/`：库存相关组件。
4. `components/canvas/`：画布相关组件。
5. `components/artwork/`：作品详情和鉴定相关组件。

拆分前应保留现有测试，避免破坏已验证行为。

## 11. 后端迁移边界

后续接后端时，以下逻辑必须迁移到服务端：

1. 挖矿产出。
2. 色块库存变更。
3. 作品鉴定。
4. 像素矩阵唯一性校验。
5. 资产生成。
6. 资产所有权。
7. 市场交易。

前端应保留：

1. UI 展示。
2. 画布交互。
3. 临时编辑状态。
4. 乐观反馈。
5. 错误提示。

## 12. 桌面客户端边界

当前不包含桌面壳。

后续如重新评估桌面客户端，应单独设计：

1. 是否使用 Tauri 或 Electron。
2. 如何复用当前 React 前端。
3. 如何处理窗口、全屏、分辨率和自动更新。
4. 如何连接后端。
5. 是否需要 Steam SDK。

桌面化不应影响当前网页 MVP 的功能验证。
